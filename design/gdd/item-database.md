# 物品数据库 (Item Database)

> **Status**: In Design
> **Author**: User + Claude
> **Last Updated**: 2026-04-07
> **Implements Pillar**: Pillar 3（资源讲故事）+ Pillar 1（压迫即乐趣）

## Overview

物品数据库（`res://data/items/`）是游戏中所有可拾取物品的唯一权威定义来源。它是一组静态的 `ItemData` 资源文件，每个文件对应一种物品，包含该物品的 ID、显示名称、堆叠上限、各楼层环境的出现权重、在电梯建造中消耗的数量比例，以及一条固定的叙事描述文本。其他系统通过物品 ID 引用数据，不在各自脚本中重新定义物品属性。玩家每次拾取、丢弃或消耗物品，本质上都是在操作指向这些定义的引用——数据库本身永远不变。

## Player Fantasy

当玩家在雾气里发现一个物品，蹲下来看到那一行描述文字——"**油渍斑斑的扳手，上面刻着某人的名字**"——他们感受到的不是游戏机制，而是这个废弃深渊曾经住着人。物品数据库负责让每一件搜刮物都有分量：不是"加 5 点资源"，而是"这是谁留下的？"

对设计师而言，每个物品都是两个旋钮同时在转：**稀缺旋钮**（权重控制玩家多难找到它）和**叙事旋钮**（描述文本控制玩家找到时感受到什么）。这个系统的成功标准是：玩家死亡后回忆起的不是"我死的时候有 3 个废铁"，而是"我死的时候差一个燃料桶就能开门"。

## Detailed Design

### Core Rules

1. **单一来源原则**：所有物品定义存放于 `res://data/items/` 目录下，每个物品一个 `.tres` 文件（Godot Resource）。其他系统通过 `item_id: String` 引用，不在各自脚本中复制属性。

2. **ItemData 资源结构**：
   ```gdscript
   class_name ItemData extends Resource

   @export var item_id: String         # 唯一标识符，例如 "scrap_metal"
   @export var display_name: String    # 显示名，例如 "废铁"
   @export var stack_limit: int        # 同类堆叠上限（1 = 不可堆叠）
   @export var spawn_weight: float     # 楼层生成时的相对权重（越高越常见）
   @export var lore_text: String       # 固定叙事描述，一行
   ```
   建造消耗数量不存储在 ItemData 中——它属于**电梯房间系统**的配方定义，ItemData 只提供物品本身的性质。

3. **堆叠规则**：同一 `item_id` 的物品在背包中共享一个槽位，数量累加，最大值为 `stack_limit`。超出上限时，超出部分不可拾取（物品留在地上）。

4. **权重表生成**：楼层生成器在生成物品时，从所有物品的 `spawn_weight` 构建加权随机池，执行加权随机选取。无显式稀有度等级——权重数值本身即是稀有度。

5. **只读，无运行时修改**：`ItemData` 资源在运行时不可修改。背包系统存储的是 `{item_id: String, count: int}` 键值对，不存储 `ItemData` 的副本。

---

**初始物品定义（MVP 5件）：**

| item_id | display_name | stack_limit | spawn_weight | lore_text |
|---------|-------------|-------------|-------------|-----------|
| `scrap_metal` | 废铁 | 10 | 40 | 锈迹斑斑的金属碎片，不知道从哪台机器上掉下来的。 |
| `canned_food` | 罐头 | 5 | 25 | 标签已经模糊，但密封圈还在。也许能吃。 |
| `old_battery` | 旧电池 | 8 | 20 | 残余电量显示 12%。足够做点什么。 |
| `first_aid_kit` | 急救包 | 3 | 10 | 里面的绷带还是折叠整齐的。有人曾经非常仔细地打包过它。 |
| `fuel_barrel` | 燃料桶 | 2 | 5 | 沉甸甸的，液体在里面晃动。电梯的心脏需要这个。 |

*权重说明：废铁最常见（40/100 ≈ 40%），燃料桶最稀有（5/100 = 5%）。总权重 = 100，便于计算期望值。*

### States and Transitions

N/A — 物品数据库是无状态的纯配置层，不存在状态机。运行时状态（数量、位置）由背包系统和楼层生成器管理。

### Interactions with Other Systems

| 下游系统 | 读取的数据 | 接口 |
|---------|-----------|------|
| 楼层生成器 | `spawn_weight`（所有物品） | 构建加权随机池，生成楼层时决定哪些物品出现 |
| 物品拾取系统 | `item_id`, `display_name`, `stack_limit` | 验证堆叠上限，向背包提交 `{item_id, count}` |
| 背包系统 | `item_id`, `stack_limit` | 查询堆叠上限，执行 add/remove 操作 |
| 电梯房间系统 | `item_id`, `display_name` | 配方中引用 item_id，UI 显示 display_name |
| 叙事文本显示 | `lore_text`, `display_name` | 拾取时或检视时展示叙事文本 |

## Formulas

物品数据库不定义公式——它是公式变量的输入端，不是公式本身。

**加权随机选取公式**（由楼层生成器实现，此处记录接口）：
```
P(item_i) = spawn_weight_i / Σ(spawn_weight_all)
```

以 MVP 5 件物品为例：

| 物品 | spawn_weight | 出现概率 |
|------|-------------|---------|
| 废铁 | 40 | 40.0% |
| 罐头 | 25 | 25.0% |
| 旧电池 | 20 | 20.0% |
| 急救包 | 10 | 10.0% |
| 燃料桶 | 5 | 5.0% |
| **合计** | **100** | **100%** |

**期望物品组合（每层 3-8 件，基准 5.5 件，来源：`FLOOR_ITEM_COUNT_MIN/MAX`）：**
- 废铁期望：5.5 × 0.40 = **2.2 件/层**
- 燃料桶期望：5.5 × 0.05 = **0.28 件/层**（约每 4 层出现 1 次）

*公式的实际实现代码属于楼层生成器 GDD，此处仅记录物品数据库提供的输入接口。*

## Edge Cases

| 场景 | 预期行为 | 理由 |
|------|---------|------|
| `item_id` 重复（两个文件定义同一 ID） | 游戏启动时断言失败，打印重复 ID 列表 | 静默覆盖会产生不可追踪的 bug |
| 拾取时背包中该物品已达 `stack_limit` | 物品留在地上，不进入背包；HUD 短暂提示"背包已满" | 不能静默丢弃，玩家必须知道 |
| `spawn_weight` = 0 | 该物品永不在普通楼层生成，但可被电梯房间系统或剧情触发器直接给予 | 保留为"特殊物品"预留接口 |
| `stack_limit` = 1 | 该物品不堆叠，每件单独占一个背包槽位 | 燃料桶等大型道具的合理限制 |
| 所有物品 `spawn_weight` = 0 | 楼层生成器权重池为空，断言失败 | 楼层必须能生成至少一件物品 |
| `lore_text` 为空字符串 | 叙事文本显示系统跳过文本显示，不渲染空文本框 | 防止 UI 出现空文本框 |

## Dependencies

| 系统 | 方向 | 依赖性质 |
|------|------|---------|
| （无上游依赖） | — | 根节点，无前置系统 |
| 背包系统 | 背包系统 → 此系统 | 硬依赖 — 读取 `stack_limit` 验证堆叠上限 |
| 物品拾取系统 | 物品拾取系统 → 此系统 | 硬依赖 — 读取 `item_id`, `display_name`, `stack_limit` |
| 楼层生成器 | 楼层生成器 → 此系统 | 硬依赖 — 读取全部物品的 `spawn_weight` 构建随机池 |
| 电梯房间系统 | 电梯房间系统 → 此系统 | 硬依赖 — 配方引用 `item_id`，UI 显示 `display_name` |
| 叙事文本显示 | 叙事文本显示 → 此系统 | 硬依赖 — 读取 `lore_text`, `display_name` |

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 调高效果 | 调低效果 |
|------|--------|---------|---------|---------|
| 废铁 `spawn_weight` | 40 | 20–60 | 更常见，背包更快填满普通材料 | 玩家材料匮乏，建造压力增大 |
| 燃料桶 `spawn_weight` | 5 | 2–15 | 电梯建造推进更快，压迫感下降 | 燃料桶过于稀缺，玩家感到沮丧而非有压力 |
| 废铁 `stack_limit` | 10 | 5–20 | 单槽携带更多，背包压力减小 | 频繁面临"扔掉哪个"的抉择，压迫感增加 |
| 燃料桶 `stack_limit` | 2 | 1–3 | 可一次携带更多，降低稀缺感 | 1 = 每件占一格，增加大件物品的空间成本 |
| 急救包 `spawn_weight` | 10 | 5–20 | 玩家更容易凑齐建造材料；目前主要影响电梯建造节奏 | 急救包成为珍贵物资，玩家更谨慎决策 |

## Visual/Audio Requirements

N/A — 物品数据库是纯数据层，无视觉或音频输出。物品的视觉表现（图标、拾取音效）属于物品拾取系统和音频系统的职责范围。

## UI Requirements

N/A — 玩家不直接与物品数据库交互。物品名称和叙事文本通过叙事文本显示系统呈现。

## Acceptance Criteria

- [ ] 所有 `.tres` 文件可从任意 GDScript 通过 `load("res://data/items/[id].tres") as ItemData` 正确加载，`item_id`, `display_name`, `stack_limit`, `spawn_weight`, `lore_text` 均非空
- [ ] 游戏启动时，若存在重复 `item_id`，日志打印错误并停止执行（开发模式断言）
- [ ] 所有 5 件 MVP 物品的 `spawn_weight` 之和 = 100（便于验算概率分布）
- [ ] 楼层生成器使用权重池随机选取时，10,000 次模拟中废铁出现概率在 38%–42% 区间（±2% 统计误差容忍）
- [ ] `stack_limit` 上限验证：向背包添加超出 `stack_limit` 的同类物品时，超出部分被拒绝，物品留在地上
- [ ] `lore_text` 为空时，叙事文本显示系统不渲染文本框（无空框 UI 残留）
- [ ] 新增一件物品只需创建一个 `.tres` 文件，无需修改其他系统代码
